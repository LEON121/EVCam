# EVCam 录制链路与存储链路说明

## 1. 文档目标

本文聚焦 EVCam 的本地核心能力，说明以下链路的实现方式：

- 摄像头初始化与预览
- 视频录制与分段
- 拍照
- 中转写入与文件转存
- 回放、分享与清理

## 2. 核心模块

| 类/模块 | 作用 |
| --- | --- |
| `MainActivity` | 主控制器，负责 UI、录制按钮、远程命令落地 |
| `camera/MultiCameraManager` | 多摄像头编排器 |
| `camera/SingleCamera` | 单摄像头封装 |
| `camera/VideoRecorder` | MediaRecorder 录制器 |
| `camera/CodecVideoRecorder` | MediaCodec + MediaMuxer 录制器 |
| `camera/EglSurfaceEncoder` | OpenGL 渲染到编码输入 Surface |
| `config/RecordingConfig` | 录制模式、分辨率、码率、帧率、水印等配置 |
| `StorageHelper` | 存储路径解析、U 盘检测、最终目录选择 |
| `FileTransferManager` | 临时目录到最终目录的异步转存 |
| `StorageCleanupManager` | 定时清理旧文件 |
| `PlaybackFragmentNew` | 视频回放 |
| `PhotoPlaybackFragmentNew` | 图片回放 |

## 3. 预览链路

### 3.1 预览初始化

`MainActivity` 会根据车型选择不同布局，并准备对应数量的 `TextureView`。

布局和车型关系大致如下：

- 银河 E5：4 路
- 银河 E5 多按钮：4 路
- 银河 L6/L7：4 路
- 银河 L7 多按钮：4 路
- 手机：2 路
- 自定义车型：1/2/4 路
- 多视角布局：自定义数量

### 3.2 MultiCameraManager 初始化

`MultiCameraManager.initCameras()` 会：

1. 创建 `SingleCamera`
2. 绑定位置：`front/back/left/right`
3. 标记重复 `cameraId` 的主从关系
4. 设置状态回调、预览尺寸回调、录制回调

### 3.3 SingleCamera 会话创建

`SingleCamera` 会根据当前场景组合不同输出 Surface：

- 主预览 `TextureView`
- 主屏悬浮窗 Surface
- 副屏 Surface
- 录制 Surface
- 鱼眼矫正中间 Surface

必要时会启用：

- Surface Sharing
- 鱼眼 GL 中间层
- 录制优先模板 `TEMPLATE_RECORD`

## 4. 录制模式

### 4.1 模式枚举

`RecordingConfig` 支持三种录制模式：

- `auto`
- `codec`
- `media_recorder`

### 4.2 自动模式判定

`RecordingConfig.shouldUseCodecRecording()` 在以下情况会切到 Codec 模式：

- 用户显式选择 `codec`
- 启用了时间水印
- 全景模式开启
- 车型是 L6/L7 系列

### 4.3 MediaRecorder 模式

实现类：`VideoRecorder`

特点：

- 基于 `MediaRecorder`
- 通过 `Surface` 接收 Camera2 画面
- 支持分段录制
- 适合更标准的硬件录制场景

### 4.4 Codec 模式

实现类：`CodecVideoRecorder`

特点：

- `MediaCodec` 编码
- `MediaMuxer` 写 MP4
- `EglSurfaceEncoder` 将相机帧渲染到编码输入 Surface
- 适合不支持 MediaRecorder 直录的车机

## 5. 视频录制链路

### 5.1 统一入口

`MultiCameraManager.startRecording()` 支持两种调用：

- 自动生成统一时间戳
- 使用指定时间戳启动录制

同时支持只录制指定摄像头集合。

### 5.2 时间戳与命名规则

视频文件命名规则：

```text
yyyyMMdd_HHmmss_{position}.mp4
```

例如：

- `20260415_142530_front.mp4`
- `20260415_142530_left.mp4`

多路同一次录制共用同一时间戳，便于后续按组聚合。

### 5.3 MediaRecorder 模式启动过程

`startMediaRecorderRecording()` 主要流程：

1. 重置首次写入通知标记。
2. 记录当前录制时间戳和启用的摄像头集合，供 Watchdog 重建使用。
3. 检查是否启用中转写入。
4. 调用 `StorageHelper.getRecordingDir(context)` 获取本次实际写入目录。
5. 根据每个摄像头的真实预览分辨率计算码率。
6. 调用 `VideoRecorder.prepareRecording(path, width, height)`。
7. 把录制 Surface 塞给 `SingleCamera`。
8. 重建 CaptureSession。
9. 等待所有会话就绪后，再统一开始录制。

### 5.4 会话超时策略

如果部分摄像头会话 3 秒内未完成配置：

- 会把未就绪的摄像头标记为失败
- 先启动已就绪摄像头的录制
- 通过 `RecordingStatusCallback` 通知上层“部分录制成功”

### 5.5 Codec 模式启动过程

`startCodecRecording()` 与 MediaRecorder 模式类似，但内部改为：

1. 为每路创建 `CodecVideoRecorder`
2. 设置时间水印、帧率、码率、分段时长等参数
3. 调用 `prepareRecording(path)` 得到编码输入链路
4. 相机帧进入 `SurfaceTexture`
5. `EglSurfaceEncoder` 渲染到编码器输入 Surface
6. 编码后由 `MediaMuxer` 输出 MP4

## 6. 分段录制

### 6.1 分段时长

支持：

- 1 分钟
- 3 分钟
- 5 分钟

统一配置来源：

- `AppConfig.getSegmentDurationMinutes()`
- `RecordingConfig.getSegmentDurationMinutes()`

### 6.2 统一时间戳提供者

`MultiCameraManager` 内部实现了统一的 `SegmentTimestampProvider`，缓存短时间内的分段时间戳，避免多摄像头切段时因为启动时机差异出现“相差 1 秒”的命名问题。

### 6.3 分段切换

分段切换时：

- `VideoRecorder` / `CodecVideoRecorder` 会在内部切换到新文件
- 回调 `onSegmentSwitch(cameraId, newSegmentIndex, completedFilePath)`
- `MultiCameraManager` 收到回调后：
  - 若启用中转写入，则把上一段文件加入异步传输
  - 重建对应摄像头 Session（MediaRecorder 模式）
  - 通知上层当前段号变化

## 7. 远程录制的特殊处理

远程录制时会调用：

- `setSegmentDurationOverride(duration + 30s)`

作用：

- 临时覆盖分段时长
- 避免远程录制任务中途按本地分段策略切段

远程录制结束后再：

- `clearSegmentDurationOverride()`

恢复用户配置值。

## 8. 首次写入与真实录制开始

`VideoRecorder` 和 `CodecVideoRecorder` 都会检测“首次数据写入”。

原因：

- 录制按钮点击成功不等于文件已开始写入
- 远程录制定时应该从真正写入时刻开始，而不是从命令收到时开始

`MultiCameraManager` 会把首次写入只向外通知一次：

- `onFirstDataWritten()`

## 9. Watchdog 与重建机制

### 9.1 触发条件

录制过程中如果出现：

- 长时间无写入
- 首次写入超时
- 编码器长时间无输出

会触发 Watchdog 重建逻辑。

### 9.2 重建策略

`MultiCameraManager.handleRecordingRebuildRequest()` 的策略是：

1. 首次重建，优先重试 MediaRecorder。
2. 如果反复失败且当前录制模式为 `auto`，则回退到 Codec 模式。
3. 重建后生成新的时间戳。
4. 通过 `timestampUpdateCallback` 通知外部更新时间戳。

这对远程录制尤其重要，因为上传时需要查找新旧时间戳对应的所有文件。

## 10. 停止录制链路

### 10.1 统一停止入口

`MultiCameraManager.stopRecording(boolean skipRelayTransfer)`

关键点：

- 立即把 `isRecording` 置为 `false`
- 后台线程执行停止操作，避免阻塞主线程
- 清理待执行启动任务与会话计数器
- 停止录制器
- 延迟清理摄像头录制 Surface

### 10.2 中转传输控制

如果满足：

- `useRelayWrite == true`
- `finalSaveDir != null`
- `skipRelayTransfer == false`

停止后会把临时目录中的剩余 MP4 文件批量加入转存队列。

`skipRelayTransfer=true` 常用于远程录制：

- 先上传
- 上传完成后再转存

## 11. 中转写入与存储路径

### 11.1 目录选择

`StorageHelper.getRecordingDir(context)` 的逻辑：

- 如果启用了中转写入，返回 `cache/temp_video`
- 否则直接返回最终视频目录

最终目录来源：

- 内部存储：`/sdcard/DCIM/EVCam_Video/`
- U 盘：`{U盘}/DCIM/EVCam_Video/`

图片目录：

- 内部存储：`/sdcard/DCIM/EVCam_Photo/`
- U 盘：`{U盘}/DCIM/EVCam_Photo/`

日志目录：

- `/sdcard/Download/EVCam_Log/`

### 11.2 U 盘检测

`StorageHelper` 通过多层机制检测外置存储：

1. 自定义路径
2. 上次检测缓存
3. `/proc/mounts`
4. `getExternalFilesDirs()`

### 11.3 中转传输

`FileTransferManager` 负责：

1. 将临时目录文件加入队列
2. 在后台线程异步移动或复制到最终目录
3. 失败时按次数重试
4. 超时清理过期临时文件

这样可以减少慢速 U 盘写入直接拖垮录制链路的风险。

## 12. 拍照链路

### 12.1 入口

- `MultiCameraManager.takePicture()`
- `SingleCamera.takePicture(timestamp, saveDelayMs)`

### 12.2 命名规则

图片命名规则：

```text
yyyyMMdd_HHmmss_{position}.jpg
```

例如：

- `20260415_142530_front.jpg`
- `20260415_142530_right.jpg`

### 12.3 保存流程

`SingleCamera.takePicture()` 当前使用预览 `TextureView.getBitmap()` 抓帧，而不是独立 `ImageReader`。

保存流程：

1. 读取当前预览帧 Bitmap
2. 调用 `StorageHelper.getPhotoDir(context)` 获取目录
3. 检查至少剩余 5MB 空间
4. 如果启用了时间角标，则先叠加水印
5. 以 JPEG 质量 90 保存

### 12.4 拍照特点

- 多路拍照时，各摄像头共用统一时间戳
- 为了减小磁盘 I/O 竞争，不同摄像头保存动作会有延迟错开

## 13. 回放链路

### 13.1 视频回放

`PlaybackFragmentNew` 会：

1. 扫描视频目录下所有 `.mp4`
2. 按时间戳聚合成 `VideoGroup`
3. 再按日期聚合成 `DateSection`

支持：

- 四宫格同步播放
- 单路放大
- 倍速
- 批量删除
- 批量分享
- 扫码互传

### 13.2 图片回放

`PhotoPlaybackFragmentNew` 与视频回放结构类似：

1. 扫描 `.jpg/.jpeg/.png`
2. 先聚合成 `PhotoGroup`
3. 再聚合为 `DateSection`

支持：

- 四宫格与单路切换
- 批量删除
- 系统分享
- 扫码互传

## 14. 分享与传输

### 14.1 系统分享

基于 `FileProvider` 对外分享文件 URI，适用于：

- 单文件分享
- 多文件分享

### 14.2 扫码互传

使用：

- `QrTransferDialog`
- `FileTransferServer`
- `QrCodeGenerator`

生成本地 HTTP 地址并展示二维码，接收端可直接浏览器访问。

HTTP 接口包括：

- `/`
- `/api/files`
- `/api/browse`
- `/download/{fileId}`

## 15. 自动清理链路

`StorageCleanupManager` 会定时清理：

- 视频目录
- 图片目录

规则：

- 冷启动 30 秒后首次检查
- 每小时检查一次
- 超过用户设置上限后，按最旧文件优先删除
- 删除时会多删 20%，避免频繁清理

特殊规则：

- 如果当前使用内部存储且剩余空间低于 3GB，会执行强制清理

## 16. 质量与性能相关配置

### 16.1 分辨率

- 默认值来自 `AppConfig` / `RecordingConfig`
- 录像时优先使用摄像头真实预览分辨率

### 16.2 帧率

支持等级：

- 标准
- 低

### 16.3 码率

支持等级：

- 低
- 标准
- 高

`RecordingConfig.calculateBitrate()` 会结合：

- 分辨率
- 帧率
- 码率等级
- 编码缩放系数

动态计算实际码率。

### 16.4 水印

支持：

- 时间水印
- GPS 水印
- 速度水印
- 地址水印
- 水印颜色
- 水印样式
- 品牌标识

当前时间水印会直接影响录制模式自动切换逻辑。

## 17. 维护建议

### 17.1 录制问题优先排查点

1. 摄像头是否已连接。
2. 预览尺寸是否已就绪。
3. 当前是否启用了中转写入。
4. U 盘是否可写。
5. 首次写入是否超时。
6. 是否触发了 Watchdog 重建。
7. 是否发生 Codec 回退。

### 17.2 存储问题优先排查点

1. `StorageHelper.getRecordingDir()` 返回的是临时目录还是最终目录。
2. `StorageHelper.getFinalVideoDir()` 是否落在预期设备上。
3. `FileTransferManager` 队列是否积压。
4. `StorageCleanupManager` 是否删除了旧文件。

### 17.3 后续优化建议

- 拍照链路可考虑迁移到 `ImageReader`，减少对 `TextureView` 的依赖。
- `MultiCameraManager` 目前承担职责很多，后续可继续拆分为：
  - 预览编排
  - 录制编排
  - Watchdog 重建
  - 中转写入调度
