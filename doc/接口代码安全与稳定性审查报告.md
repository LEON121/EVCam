# EVCam 接口代码安全与稳定性审查报告

## 1. 审查范围

本次审查基于现有文档与源码，重点覆盖以下三类接口链路：

- 远程控制链路：钉钉、Telegram、飞书、`WakeUpHelper`、`RemoteCommandDispatcher`
- 本地传输链路：二维码互传、本地 HTTP 文件服务
- 录制与联动链路：录制入口、补盲服务控制、日志上传、配置持久化

说明：

- 本报告为静态代码审查结果，不包含真实车机环境压测、网络渗透测试或 APK 逆向验证。
- 风险等级分为：`严重`、`高`、`中`、`低`。

## 2. 总结结论

当前接口实现存在 2 个严重问题、2 个高风险问题、2 个中风险稳定性问题。  
优先级建议：

1. 先修复导出入口与本地 HTTP 服务的未授权访问问题。
2. 再处理日志上传明文传输、凭证明文存储与备份问题。
3. 最后处理远程录制并发竞态和 4 路摄像头写死等待问题。

## 3. 详细发现

### 3.1 严重：导出的 `MainActivity` 直接信任 `remote_action`，任意第三方 App 可伪造远程命令

**证据**

- `app/src/main/AndroidManifest.xml:82-90`
  - `MainActivity` 为 `android:exported="true"`，且作为启动页暴露给系统和外部应用。
- `app/src/main/java/com/kooo/evcam/MainActivity.java:594-817`
  - `handleRemoteCommandFromIntent()` 直接读取 `Intent` 中的 `remote_action`、`remote_source`、`remote_duration` 等参数。
- `app/src/main/java/com/kooo/evcam/MainActivity.java:823-845`
  - `executeRemoteCommand()` 可直接触发：
    - 远程录制
    - 远程拍照
    - 启动持续录制
    - 停止录制并退后台

**影响**

- 任意本机第三方 App 都可以显式启动 `MainActivity`，并附带：
  - `remote_action=start_recording`
  - `remote_action=stop_recording`
  - `remote_action=record`
  - `remote_action=photo`
- 这会绕过钉钉 / Telegram / 飞书的鉴权体系，直接触发录制、拍照、前后台切换等高权限行为。

**建议**

- 不要直接信任 `exported` Activity 中的远程控制参数。
- 增加至少一种校验机制：
  - 自定义签名级权限
  - 仅允许应用内 `PendingIntent`
  - 一次性 token / nonce
  - 校验调用方 UID / 包名
- 更稳妥的做法是：
  - `MainActivity` 只负责 UI
  - 远程动作由 `exported=false` 的内部组件或 `BroadcastReceiver` 处理

---

### 3.2 严重：本地文件互传 HTTP 服务允许未授权浏览和下载任意可读文件

**证据**

- `app/src/main/java/com/kooo/evcam/transfer/FileTransferServer.java:106-127`
  - 暴露 `/api/browse` 与 `/download/{fileId}`。
- `app/src/main/java/com/kooo/evcam/transfer/FileTransferServer.java:834-975`
  - `serveBrowseRequest(path)` 直接按请求参数浏览任意路径，未做目录白名单限制。
- `app/src/main/java/com/kooo/evcam/transfer/FileTransferServer.java:1156-1164`
  - `serveFile(fileId)` 如果 `pendingFiles` 中找不到，就把 `fileId` 当作真实路径直接读取。
- `app/src/main/java/com/kooo/evcam/transfer/FileTransferServer.java:145-148`
  - 对所有响应加了 `Access-Control-Allow-Origin: *`。

**影响**

- 只要同一局域网内有人访问这个服务，就可以：
  - 枚举设备可读目录
  - 下载任何应用进程可读文件
  - 结合路径猜测读取应用私有数据、配置文件、日志等
- CORS 全开放意味着恶意网页也可以借助浏览器直接读取该服务返回的数据。

**建议**

- `/api/browse` 必须限制在明确白名单目录中，例如：
  - 当前待分享文件目录
  - `DCIM/EVCam_Video`
  - `DCIM/EVCam_Photo`
- `/download/{fileId}` 只允许下载 `pendingFiles` 中注册过的文件，禁止把参数当真实路径解析。
- 增加访问控制：
  - 临时随机 token
  - 一次性会话码
  - 仅允许局域网首次扫码端访问
- 非必要不要返回 `Access-Control-Allow-Origin: *`。

### 3.3 高：日志上传使用硬编码 HTTP 服务地址和固定 Token，存在泄露与滥用风险

**证据**

- `app/src/main/java/com/kooo/evcam/AppLog.java:49-50`
  - 硬编码 `GOTIFY_SERVER_URL` 与 `GOTIFY_APP_TOKEN`
- `app/src/main/java/com/kooo/evcam/AppLog.java:333-445`
  - 日志通过 `HttpURLConnection` 上传
- `app/src/main/AndroidManifest.xml:60`
  - `android:usesCleartextTraffic="true"`

**影响**

- 日志内容包含：
  - 设备型号
  - App 版本
  - 用户自定义问题描述
  - 整段运行日志
- 由于是明文 HTTP：
  - 同网段或中间网络设备可窃听日志内容
  - 请求可被篡改
- 硬编码 Token 一旦 APK 外泄，任何人都可以伪造日志消息或滥用该 Gotify 通道。

**建议**

- 至少改为 HTTPS。
- 不要把服务 Token 硬编码在客户端。
- 采用服务端签发短期上传凭证，或让用户显式配置上传目标。
- 上传前增加日志脱敏，过滤 access token、bot token、secret 等敏感字段。

### 3.4 高：远程平台凭证明文保存在 `SharedPreferences`，且应用允许备份与设备迁移

**证据**

- `app/src/main/java/com/kooo/evcam/dingtalk/DingTalkConfig.java:24-47`
  - 保存 `client_secret`、`access_token`
- `app/src/main/java/com/kooo/evcam/telegram/TelegramConfig.java:28-43`
  - 保存 `bot_token`
- `app/src/main/java/com/kooo/evcam/feishu/FeishuConfig.java:29-44`
  - 保存 `app_secret`、`access_token`
- `app/src/main/AndroidManifest.xml:52-55`
  - `android:allowBackup="true"`
- `app/src/main/res/xml/backup_rules.xml:8-13`
  - 未排除共享配置
- `app/src/main/res/xml/data_extraction_rules.xml:6-18`
  - 也未排除共享配置或设备迁移

**影响**

- 远程机器人凭证与 access token 会以明文形式落在应用私有配置中。
- 在允许备份/迁移的情况下，这些配置存在被：
  - 设备迁移
  - 云备份
  - 调试备份
  - Root 取证
 读取的风险。

**建议**

- 敏感配置迁移到：
  - `EncryptedSharedPreferences`
  - Android Keystore 包装的安全存储
- 在 `backup_rules.xml` / `data_extraction_rules.xml` 中显式排除：
  - `dingtalk_config`
  - `telegram_config`
  - `feishu_config`
  - `heartbeat_config`
- 如果没有强备份需求，可直接关闭 `allowBackup`。

### 3.5 中：远程录制状态切换缺少原子保护，多平台并发命令下存在竞态

**证据**

- `app/src/main/java/com/kooo/evcam/remote/handler/RemoteCommandHandler.java:39-46`
  - `isRemoteRecording`、`isPreparingRecording`、`currentContext` 只是 `volatile` / 普通字段。
- `app/src/main/java/com/kooo/evcam/remote/handler/RemoteCommandHandler.java:122-173`
  - `startRemoteRecording()` 先判断 `if (isRemoteRecording)`，稍后才写入 `isRemoteRecording = true`，中间没有锁或 CAS。

**影响**

- 如果钉钉、Telegram、飞书几乎同时发起录制命令，可能出现：
  - 两个命令同时通过“未在远程录制”的判断
  - `currentContext` 被后一个任务覆盖
  - 自动停止定时器、上传对象、恢复手动录制逻辑相互干扰
- 这类问题很难复现，但一旦出现，行为会非常不稳定。

**建议**

- 用 `AtomicBoolean.compareAndSet(false, true)` 或显式锁把“判断 + 置位”做成原子操作。
- `currentContext`、`pendingDurationSeconds`、`autoStopRunnable` 也建议在同一临界区更新。
- 增加并发命令单元测试或集成测试。

---

### 3.6 中：通用远程意图链路把“就绪摄像头数量”写死为 4，非 4 路车型会引入额外等待和时序抖动

**证据**

- `app/src/main/java/com/kooo/evcam/MainActivity.java:799-815`
  - 通用远程命令分支中使用 `connectedCount < 4` 作为等待条件。

**影响**

- 在手机、1 路自定义、2 路自定义、多视角非 4 路布局下：
  - 远程命令每次都会额外等待
  - 日志中持续出现“摄像头不足 4 个”的误导性警告
  - 某些边界时序下会把本来已就绪的设备误判为“未准备好”

**建议**

- 等待条件应改为：
  - `configuredCameraCount`
  - 或 `appConfig.getCameraCount()`
  - 或至少 `cameraManager.hasConnectedCameras()`
- 不同车型应该使用统一的“目标摄像头数量”来源，而不是在远程链路中硬编码。

## 4. 建议整改顺序

### 第一优先级

- 修复 `MainActivity` 导出入口被伪造远程命令的问题
- 修复 `FileTransferServer` 任意路径浏览/下载问题

### 第二优先级

- 移除日志上传中的明文 HTTP 与硬编码 Token
- 调整远程平台凭证存储和备份策略

### 第三优先级

- 给 `RemoteCommandHandler` 增加并发保护
- 修复远程命令链路的 4 路摄像头硬编码

## 5. 额外建议

- 为本地 HTTP 文件服务增加最基本的审计日志和访问 token。
- 为远程控制链路增加统一的安全边界设计说明，避免后续继续走 `Intent Extra` 直接控业务。
- 为以下场景补测试：
  - 多平台同时发起远程录制
  - 手机/2 路/自定义车型的远程录制
  - U 盘拔插期间的录制与转存
  - 文件互传服务在不同浏览器和局域网环境下的访问控制
