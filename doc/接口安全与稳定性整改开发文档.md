# EVCam 接口安全与稳定性整改开发文档

## 1. 文档目的

本文档用于指导 `接口代码安全与稳定性审查报告` 中问题的整改开发工作，目标是：

- 修复高风险安全漏洞
- 提升远程控制与本地文件传输的稳定性
- 降低后续维护成本
- 为测试、验收和发布提供统一依据

## 2. 整改范围

本次整改覆盖以下 6 项问题：

1. `MainActivity` 导出入口可被伪造远程命令调用
2. 本地 HTTP 文件服务支持任意路径浏览/下载
3. 日志上传使用明文 HTTP 和硬编码 Token
4. 远程平台凭证明文存储且可被备份/迁移
5. 远程录制状态切换存在并发竞态
6. 远程命令链路将摄像头就绪数量写死为 4

## 3. 改造目标

### 3.1 安全目标

- 外部 App 无法伪造远程命令直接控制录制/拍照
- 本地传输服务只能访问授权目录和授权文件
- 敏感配置不再明文备份、不再明文上传
- 日志上传链路支持 HTTPS 与敏感信息脱敏

### 3.2 稳定性目标

- 同时收到多平台远程录制命令时，行为可预测
- 非 4 路车型下远程控制不再额外等待和误判
- 中转写入、上传、停止录制之间的边界更清晰

## 4. 总体设计方案

## 4.1 远程命令入口收敛

### 问题

当前 `MainActivity` 为 `exported=true`，且直接解析 `remote_action`。

### 方案

- `MainActivity` 保留启动页能力，但不再直接信任外部传入的远程控制参数。
- 新增“内部远程命令网关”，仅允许应用内组件调用。
- 推荐实现方式：
  - 新增内部 `BroadcastReceiver` 或 `Service`，设置 `exported=false`
  - 由 `WakeUpHelper` 改为向该内部组件发送命令
  - 内部组件再转发给 `MainActivity` 或直接分发到 `RemoteCommandDispatcher`
- 如果必须继续使用 Activity Intent，则必须加签名级权限或一次性 token 校验。

### 涉及文件

- `app/src/main/AndroidManifest.xml`
- `app/src/main/java/com/kooo/evcam/MainActivity.java`
- `app/src/main/java/com/kooo/evcam/WakeUpHelper.java`
- 新增内部命令网关类

## 4.2 本地 HTTP 文件服务收口

### 问题

`FileTransferServer` 当前支持任意路径浏览和按真实路径下载。

### 方案

- 移除“任意路径下载”逻辑：
  - `serveFile()` 只允许下载 `pendingFiles` 中注册过的文件
  - 禁止 `new File(fileId)` 直接作为真实路径解析
- 对目录浏览增加白名单：
  - 仅允许 `EVCam_Video`
  - `EVCam_Photo`
  - 当前待传文件目录
- 增加会话级 token：
  - 启动服务时生成随机 token
  - `/api/files`、`/api/browse`、`/download/*` 必须带 token
- 缩小 CORS 范围：
  - 默认关闭跨域
  - 如保留网页端访问，仅允许同源

### 涉及文件

- `app/src/main/java/com/kooo/evcam/transfer/FileTransferServer.java`
- `app/src/main/java/com/kooo/evcam/transfer/QrTransferDialog.java`

## 4.3 日志上传整改

### 问题

- 使用明文 HTTP
- 客户端硬编码 Gotify Token
- 日志可能包含敏感字段

### 方案

- 日志上传改为 HTTPS
- 移除客户端硬编码 Token
- 服务端改为：
  - 下发临时上传凭证
  - 或由用户配置上传地址与凭证
- 客户端增加日志脱敏过滤：
  - `bot_token`
  - `access_token`
  - `client_secret`
  - `app_secret`
  - 其他常见鉴权字段

### 涉及文件

- `app/src/main/java/com/kooo/evcam/AppLog.java`
- `app/src/main/AndroidManifest.xml`

## 4.4 敏感配置安全存储

### 问题

远程平台密钥、access token 明文保存在 `SharedPreferences`，且允许备份。

### 方案

- 敏感配置迁移到 `EncryptedSharedPreferences`
- 基于 `MasterKey` / Android Keystore 保存主密钥
- 为旧配置增加迁移逻辑：
  - 首次读取时检测旧偏好
  - 迁移成功后清除旧明文字段
- 在备份规则中排除敏感配置：
  - `dingtalk_config`
  - `telegram_config`
  - `feishu_config`
  - `heartbeat_config`

### 涉及文件

- `app/src/main/java/com/kooo/evcam/dingtalk/DingTalkConfig.java`
- `app/src/main/java/com/kooo/evcam/telegram/TelegramConfig.java`
- `app/src/main/java/com/kooo/evcam/feishu/FeishuConfig.java`
- `app/src/main/res/xml/backup_rules.xml`
- `app/src/main/res/xml/data_extraction_rules.xml`
- `app/build.gradle.kts`（如需引入安全存储依赖）

## 4.5 远程录制并发保护

### 问题

`RemoteCommandHandler.startRemoteRecording()` 中“判断是否在录制”和“设置录制中”不是原子操作。

### 方案

- 使用 `AtomicBoolean.compareAndSet(false, true)` 控制录制入口
- 或增加 `synchronized` 临界区，统一保护以下字段：
  - `isRemoteRecording`
  - `isPreparingRecording`
  - `currentContext`
  - `autoStopRunnable`
  - `pendingDurationSeconds`
- 禁止不同平台在同一时间并发创建远程录制任务

### 涉及文件

- `app/src/main/java/com/kooo/evcam/remote/handler/RemoteCommandHandler.java`

## 4.6 车型摄像头数量适配

### 问题

远程意图执行链路中将摄像头就绪数量写死为 `4`。

### 方案

- 统一改为读取动态目标值：
  - `configuredCameraCount`
  - 或 `appConfig.getCameraCount()`
- Telegram、飞书、钉钉三条分支保持一致
- 等待逻辑改成：
  - 如果目标数量未知，则以 `hasConnectedCameras()` 为底线
  - 如果目标数量已知，则按车型数量判断

### 涉及文件

- `app/src/main/java/com/kooo/evcam/MainActivity.java`

## 7. 测试方案

## 7.1 功能测试

- 钉钉远程录制/拍照/状态/前后台切换
- Telegram 远程录制/拍照/状态
- 飞书远程录制/拍照/状态
- 本地扫码互传页面加载、文件下载、目录浏览
- 手机模式、2 路模式、4 路模式远程录制

## 7.2 安全测试

- 外部 App 显式启动 `MainActivity` 附带 `remote_action`，验证无效
- 局域网直接访问旧 `/api/browse` 和 `/download/绝对路径`，验证被拒绝
- 检查 APK 中不再出现硬编码日志上传 Token
- 检查应用数据备份中不包含机器人凭证

## 7.3 稳定性测试

- 同时从钉钉和 Telegram 发起远程录制
- 连续快速发起多次远程拍照
- 远程录制过程中切换前后台
- 录制中 U 盘拔插
- 非 4 路车型重复触发远程命令

## 8. 验收标准

- 非本应用内部链路无法通过 Intent 直接触发远程录制/拍照
- 文件互传服务不能浏览和下载白名单外路径
- 所有远程平台凭证迁移到加密存储
- 备份规则中明确排除敏感配置
- 日志上传走 HTTPS，且无硬编码服务 Token
- 并发远程录制时只能成功进入一个任务
- 手机和自定义车型远程录制无额外 4 路等待

## 9. 回滚方案

- 每个阶段独立提交，避免一次性大改
- 文件互传服务可先保留旧 UI，但新后端接口优先上线
- 安全存储改造必须带迁移逻辑，避免升级后配置丢失
- 如 HTTPS 服务端未准备好，可先保留上传开关，但默认关闭旧上传入口
