# EVCam 远程控制接口说明

## 1. 目标与范围

本文说明 EVCam 当前已实现的远程控制能力，包括钉钉、Telegram、飞书三类平台的：

- 配置入口
- 启动方式
- 命令集合
- 唤醒与执行链路
- 上传链路
- 关键限制与注意事项

远程控制的核心目标是：在应用退到后台甚至车机息屏后，仍能通过机器人命令唤醒应用、完成录制/拍照、上传结果并返回状态。

## 2. 模块结构

### 2.1 主要类

| 类/包 | 作用 |
| --- | --- |
| `RemoteViewFragment` | 钉钉配置页面 |
| `TelegramFragment` | Telegram 配置页面 |
| `FeishuFragment` | 飞书配置页面 |
| `RemoteServiceManager` | 远程服务单例，跨 Activity 持有服务实例 |
| `remote/RemoteCommandDispatcher` | 统一的远程命令分发入口 |
| `remote/handler/RemoteCommandHandler` | 平台无关的远程录制/拍照公共逻辑 |
| `remote/handler/DingTalkHandler` | 钉钉适配层 |
| `remote/handler/TelegramHandler` | Telegram 适配层 |
| `remote/handler/FeishuHandler` | 飞书适配层 |
| `WakeUpHelper` | 后台唤醒 MainActivity 并传递远程动作 |
| `MediaFileFinder` | 根据时间戳查找视频/图片文件 |
| `MediaUploadService` | 平台上传服务统一接口 |

### 2.2 平台实现

- 钉钉：`dingtalk/DingTalkStreamManager`
- Telegram：`telegram/TelegramBotManager`
- 飞书：`feishu/FeishuBotManager`

三者都实现了“命令接收 -> 唤醒应用 -> 执行业务 -> 上传结果 -> 回复状态”的完整闭环。

## 3. 配置入口

### 3.1 钉钉

配置保存在 `dingtalk_config`：

- `client_id`
- `client_secret`
- `access_token`
- `token_expire_time`
- `auto_start`

页面入口：`RemoteViewFragment`

支持动作：

- 保存配置
- 测试连接
- 启动/停止服务
- 自动启动开关

### 3.2 Telegram

配置保存在 `telegram_config`：

- `bot_token`
- `allowed_chat_ids`
- `bot_api_host`
- `last_update_id`
- `auto_start`

页面入口：`TelegramFragment`

说明：

- `allowed_chat_ids` 为空时默认允许所有会话。
- `bot_api_host` 允许自定义反向代理地址，默认 `https://api.telegram.org`。

### 3.3 飞书

配置保存在 `feishu_config`：

- `app_id`
- `app_secret`
- `allowed_user_ids`
- `access_token`
- `token_expire_time`
- `auto_start`

页面入口：`FeishuFragment`

说明：

- `allowed_user_ids` 为空时默认允许所有用户。

## 4. 服务生命周期

### 4.1 前台页面启动

`MainActivity` 可直接启动三类远程服务：

- `startDingTalkService()`
- `startTelegramService()`
- `startFeishuService()`

启动后服务实例会注册到 `RemoteServiceManager`，避免 Activity 重建时服务中断。

### 4.2 从前台服务拉起

`RemoteServiceManager.startRemoteServicesFromService(Context)` 支持在 `CameraForegroundService` 等后台场景下自动启动已配置且启用了 `auto_start` 的远程服务。

这样可以做到：

- Activity 被回收后，远程服务仍尽量保持运行。
- 收到命令时，不要求界面已经在前台。

### 4.3 状态提供

`RemoteServiceManager` 允许 `MainActivity` 注册 `StatusInfoProvider`。

状态信息有两种来源：

- 优先使用 `MainActivity.buildStatusInfo()`
- 如果 Activity 不可用，则回落到 `RemoteServiceManager.buildBasicStatusInfo()`

## 5. 命令接口

### 5.1 钉钉命令

命令来源：

- `DingTalkStreamManager`
- 兼容旧实现 `DingTalkBotMessageListener`、`DingTalkCommandReceiver`

支持命令：

| 命令 | 说明 |
| --- | --- |
| `状态` / `status` | 返回应用状态 |
| `前台` / `foreground` | 将应用切到前台 |
| `后台` / `background` | 将应用切到后台 |
| `启动录制` / `开始录制` / `start` | 开始持续录制 |
| `结束录制` / `停止录制` / `stop` | 停止持续录制 |
| `录制` / `record` | 录制 60 秒视频 |
| `录制30` / `record 30` | 录制指定秒数视频 |
| `拍照` / `photo` | 远程拍照 |
| `退出` / `exit` | 请求退出 |
| `确认退出` | 执行退出 |
| `帮助` / `help` | 返回帮助信息 |

### 5.2 Telegram 命令

支持命令：

| 命令 | 说明 |
| --- | --- |
| `/status`、`状态` | 返回状态 |
| `/foreground`、`前台` | 切到前台 |
| `/background`、`后台` | 切到后台 |
| `/start_rec`、`启动录制`、`开始录制`、`start` | 持续录制 |
| `/stop_rec`、`结束录制`、`停止录制`、`stop` | 停止持续录制 |
| `/record`、`/record 30`、`录制`、`录制30` | 指定秒数录制 |
| `/photo`、`拍照` | 拍照 |
| `/exit`、`退出` | 请求退出 |
| `/confirm_exit`、`确认退出` | 执行退出 |
| `/help`、`帮助`、`/start` | 帮助 |

录制秒数解析规则：

- 默认 60 秒
- 最小 5 秒
- 最大 600 秒

### 5.3 飞书命令

支持命令：

## 6. 唤醒与执行链路

### 6.1 唤醒参数

`WakeUpHelper` 通过启动 `MainActivity` 并带参数实现后台唤醒。

关键参数：

- `remote_action`
- `remote_wake_up`
- `remote_source`
- `remote_duration`
- 钉钉：`remote_conversation_id`、`remote_conversation_type`、`remote_user_id`
- Telegram：`telegram_chat_id`
- 飞书：`feishu_chat_id`、`feishu_message_id`

### 6.2 典型时序

#### 远程录制

1. 机器人接收命令。
2. 平台管理器解析命令与录制时长。
3. 先回复“开始录制中”。
4. 通过 `WakeUpHelper` 唤醒 `MainActivity`。
5. `MainActivity` 将命令转给 `RemoteCommandDispatcher`。
6. `RemoteCommandHandler.startRemoteRecording()` 执行：
   - 检查是否已有远程录制任务
   - 检查摄像头控制器是否可用
   - 检查是否有已连接摄像头
   - 生成统一时间戳
   - 如当前有手动录制，则先停手动录制
   - 设置分段时长覆盖，避免远程录制中途自动分段
   - 调用摄像头控制器开始录制
7. 首次写入数据后，通过 `onFirstDataWritten()` 启动远程录制定时器。
8. 定时器到时后停止录制、上传视频、必要时恢复手动录制。

#### 远程拍照

1. 机器人接收拍照命令。
2. 平台管理器先回复“正在拍照”。
3. 唤醒 `MainActivity`。
4. `RemoteCommandHandler.startRemotePhoto()` 调用拍照。
5. 等待约 5 秒后查找对应图片并上传。

## 7. 统一远程处理逻辑

`RemoteCommandHandler` 负责 90% 的通用流程，平台差异只放在适配层。

公共能力包括：

- 远程录制状态管理
- 准备中状态管理
- 自动停止计时
- 与手动录制的互斥和恢复
- 视频/图片查找
- 上传成功/失败回调
- 远程唤醒后自动退后台

这意味着新增远程平台时，优先复用：

- `ChatIdentifier`
- `RemotePlatform`
- `RemoteCommandHandler`
- `MediaUploadService`

## 8. 上传链路

### 8.1 文件查找

`MediaFileFinder` 的查找顺序：

- 先查临时目录 `cache/temp_video`
- 再查最终视频目录 `StorageHelper.getVideoDir(context)`

视频查找支持：

- 单一时间戳
- 多时间戳列表

多时间戳场景用于 Watchdog 重建录制后，把新旧时间戳对应的视频一起找出来上传。

图片查找规则：

- 目录：`StorageHelper.getPhotoDir(context)`
- 命名：`{timestamp}_{position}.jpg`

### 8.2 上传接口

统一接口：

```java
void uploadVideos(List<File> videoFiles, ChatIdentifier chatId, RemoteUploadCallback callback)
void uploadPhotos(List<File> photoFiles, ChatIdentifier chatId, RemoteUploadCallback callback)
```

### 8.3 平台限制

- Telegram：Bot API 上传文件不能超过 50MB。
- 飞书：上传文件超过 30MB 时会失败。
- 钉钉：没有在统一层写死大小限制，但仍受平台接口和网络环境影响。

### 8.4 错误处理

- `RemoteCommandHandler.handleUploadError()` 允许平台扩展错误提示。
- Telegram / 飞书会针对超限错误返回中文说明。
- 即使上传失败，中转写入模式下也会尽量把文件转移到最终目录，避免录制文件丢失。

## 9. 状态接口

### 9.1 文本状态

`MainActivity.buildStatusInfo()` 返回远程状态文本，包含：

- 是否录制
- 是否远程录制
- 当前录制时长与分段编号
- 已连接摄像头数
- 存储位置与剩余空间
- 应用前后台状态
- 自动录制、心跳推图、分段时长、车型等摘要

### 9.2 基础状态

当 `MainActivity` 不可用时，`RemoteServiceManager.buildBasicStatusInfo()` 至少返回：

- 三类远程服务在线情况
- 当前存储剩余空间

## 10. 前后台切换与退出

### 10.1 切到前台

- 调用 `WakeUpHelper.launchForForeground()`
- 返回文本：应用已切换到前台

### 10.2 切到后台

- 不直接再启动 Activity
- 使用广播 `com.kooo.evcam.ACTION_MOVE_TO_BACKGROUND`
- 由 `MainActivity` 收到后执行 `moveTaskToBack(true)`

### 10.3 退出应用

退出采用二次确认：

1. 发送 `退出`
2. 发送 `确认退出`

执行后会：

- 停止远程服务
- 调用 `MainActivity.exitApp()`

## 11. 设计注意事项

### 11.1 后台限制

Android 车机场景下，后台直接启动 Activity 受系统限制，因此必须配合：

- 悬浮窗权限
- WakeLock
- 前台服务

### 11.2 远程录制与手动录制冲突

当前策略：

- 如果收到远程录制时本地正在手动录制，先暂停手动录制
- 远程录制完成后再尝试恢复手动录制

### 11.3 计时起点

远程录制定时不是从命令接收瞬间开始，而是从“首次有录制数据写入”开始。这样更接近用户期望的有效录制时长。

### 11.4 新平台接入建议

新增平台时建议只做三件事：

1. 新建配置类与配置页面。
2. 新建平台管理器接收命令。
3. 新建 `RemoteCommandHandler` 子类和对应上传适配器。

公共逻辑尽量不要再复制到新平台代码中。
